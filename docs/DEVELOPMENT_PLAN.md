# План разработки: мониторинг ДГУ (MVP / демо)

## Цели и стек

- **Laravel 13**, **SQLite**, **Blade + Tailwind** (Breeze), **Spatie Permission** (роль администратора).
- Телеметрия от **симулятора** (REST JSON), **без** отдельного Device Gateway и **без** Swagger на первом этапе.
- Связь «живая» если пакет не старше **10 минут**; признак **долго offline** — **12+ часов** без телеметрии (отдельный бейдж/фильтр).

## Фазы

### Фаза 0 — каркас (текущий спринт)

- Репозиторий Laravel + Breeze + SQLite.
- Миграции: `dgus`, `telemetry_snapshots`, `global_thresholds`, `alerts`, `alert_events`, `app_notifications`, `audit_logs`, `system_logs`, поля пользователя (`phone`, `job_title`).
- Каталоги по разделам: `app/Features/{Dashboard,Dgu,Telemetry,Alerts,Settings,...}` (контроллеры, действия, политики).
- Роли: `admin`; сидер администратора и глобальных порогов по эталону АД100С.
- **Ingest API**: `POST /api/v1/telemetry` + заголовок токена; хеш токена в `dgus.telemetry_token_hash`.
- **Artisan**: `php artisan dgu:simulate-telemetry` для демо.
- **Dashboard**: KPI — всего ДГУ, активно/неактивно по связи, активные тревоги.
- **Health**: встроенный `/up` + при необходимости расширенный JSON.

### Фаза 1 — CRUD и карточка ДГУ

- Список/создание/редактирование ДГУ (только admin): поля из ТЗ, показ **токена приёма** один раз при создании.
- Карточка ДГУ: таблица параметров по slug, статусы норма/отклонение, график (шаг **1 мин**, окно по умолчанию **24 ч**).

### Фаза 2 — Тревоги и уведомления

- Расчёт тревог из глобальных порогов при приёме телеметрии.
- Подтверждение тревоги (только admin): комментарий, запись в `alert_events` (эскалации/история).
- In-app уведомления + **дублирование email** (очередь + шаблоны, SMTP из `.env`).

### Фаза 3 — Управление, карта, настройки

- Старт/стоп (демо: два состояния + аудит); заготовки под режимы/расписания/ручной ввод.
- Карта: по одному маркеру на ДГУ (Leaflet + OSM по умолчанию).
- Пользователи: приглашение по email (Laravel notifications + ссылка сброса пароля или invite flow).
- Разделы «Уведомления», «Логи», «Аудит» — отдельные представления с точностью до секунды.

### Фаза 4 — Качество

- PHPUnit/Pest по критичным сервисам (ingest, расчёт тревог, права).
- Pint, PHPStan **level 5** (дефолт), pre-commit по согласованию.
- Ретеншн сырых точек и параметры нагрузки — в `.env` (демо: до ~100 ДГУ, интервал 1 мин, хранение N дней).

## Порядок веток / итераций

1. `main` — только каркас + миграции + сиды + ingest + симулятор + KPI dashboard.
2. Затем ветки по фазам 1–3 параллельно по возможности (карта и email независимы от части CRUD).

## Переменные окружения (минимум)

- `DB_CONNECTION=sqlite`
- `TELEMETRY_STALE_MINUTES=10`, `TELEMETRY_LONG_OFFLINE_HOURS=12`
- `TELEMETRY_RETENTION_DAYS=14` (пример)
- Почта: `MAIL_*` для дублирования тревог.


## Статус (итерация)

- CRUD ДГУ с политикой: просмотр/список — любой авторизованный; создание/правка/удаление — только `admin`.
- Карточка ДГУ: последний снимок, таблица параметров vs глобальные пороги, график 24ч (Chart.js), индикаторы связи 10 мин / долго offline 12 ч.
- Глобальные пороги: список и редактирование под `role:admin`.
- Тревоги: список открытых, карточка, подтверждение с комментарием (только admin), аудит `alert.acknowledged`.
- Навигация: ДГУ, Тревоги, Пороги (для admin).
- Базовый `Controller` расширяет `Illuminate\Routing\Controller` + `AuthorizesRequests` для `authorizeResource`.
